PUNTO 2
¿Cumplen con sus funciones “básicas”? 
Cada uno de los controller, service, repository y models cumplen con lo que vendría a 
ser su función “original”. Los controller reciben las peticiones del cliente (HTTP), 
manejan GET, POST, PUT, DELETE, llaman al service para que haga el trabajo y 
devuelven la respuesta. Los service contienen la lógica del programa, deciden qué 
hacer con los datos y usan los repositories para acceder a la base de datos. Tiene 
“funciones” como calcular promedio, validar algo, etc. Los repository se comunican 
con la base de datos y hacen consultas (guardar, buscar, borrar). Puede usar ciertos 
metodos como “findById(), save() y delete()”. Y, por último, los models representan las 
tablas de la base de datos con los respectivos atributos de las tablas. 

¿Cumplen las funcionalidades dadas en el TP?

4) Se revisaron los endpoints del sistema para identificar aquellos que estaban devolviendo directamente las entidades del modelo. Esto no es una buena práctica, ya que las entidades pueden incluir más información de la necesaria o incluso relaciones con otras tablas de la base de datos que el cliente no necesita recibir. Para solucionar esto se implementaron DTOs (Data Transfer Objects), que permiten enviar únicamente los datos relevantes al cliente.
En el caso del endpoint GET /api/obras, originalmente el método del controller devolvía una lista de objetos de tipo Obra. Esto implicaba que se enviaban todos los atributos de la entidad, incluyendo relaciones con otras entidades como Artista, Sala y TipoObra. Para evitar esto, se modificó el método para que en lugar de devolver Obra devuelva una lista de ObraDTO. Antes de enviar la respuesta, cada objeto Obra se convierte en un ObraDTO, conservando solamente los campos necesarios: idObra, nombre, descripcion, epoca y nivelPopularidad. De esta manera se evita exponer información interna que no es necesaria para el cliente.
El mismo cambio se aplicó en el endpoint GET /api/obras/{id}. En la versión original se devolvía directamente el objeto Obra obtenido desde el servicio. Luego de la modificación, el controller crea un objeto ObraDTO utilizando los datos principales de la obra y devuelve ese DTO como respuesta. Esto permite mantener el mismo criterio utilizado en el resto de los endpoints y asegurar que solo se envíe la información necesaria.
También se realizaron cambios en los endpoints relacionados con los eventos. En el endpoint GET /api/eventos/{id}, inicialmente se devolvía la entidad Evento completa. Esto se reemplazó por un EventoDTO, que contiene únicamente los campos idEvento, nombre y descripcion. Para lograrlo, el controller obtiene el objeto Evento desde el servicio y luego genera un EventoDTO con los datos que se desean enviar en la respuesta.
Finalmente, en el endpoint GET /api/eventos también se estaba devolviendo una lista de entidades Evento. Luego de la modificación, el método transforma cada objeto Evento en un EventoDTO y devuelve una lista de estos DTOs.
El uso de DTOs permite separar la estructura de la API del modelo interno de la aplicación. Además, evita enviar información innecesaria al cliente y hace que las respuestas sean más claras y livianas. Esto mejora la organización del código y facilita el mantenimiento de la aplicación a futuro.


5) CAMBIOS QUE HICIMOS EN LOS CONTROLLER: 
En las clases se modificó la forma en la que se devuelven los errores cuando hay problemas con las fechas ingresadas. Antes, el código devolvía los errores utilizando headers HTTP, lo cual no es lo más claro para el cliente. Se cambió para que los errores se devuelvan utilizando la clase ErrorResponse (que fue creada por nosotros como un extra para manejar los mensajes de una manera más facil), que genera un mensaje en formato JSON. De esta manera, si el usuario envía una fecha con un formato incorrecto u ocurre algún error al procesar la consulta, el sistema responde con un HTTP 400 (Bad Request) y un mensaje explicando el problema.  Esto se hizo para evitar que se busque sobre un id que no existe o si el mensaje JSON enviado en un post está vacío.

Punto 4:


"Cómo usar DTO en un controller de Spring Boot"

Respuesta:
Un DTO (Data Transfer Object) en un controller de Spring Boot se usa para recibir o enviar datos sin exponer directamente la entidad del modelo.
La idea es separar:
    • Entidad (model) → representa la tabla en la base de datos
    • DTO → representa los datos que viajan en la API


"Como evito las entidades JPA en una API REST"

Para evitar exponer entidades JPA en una API REST, en Spring Boot se usa normalmente el patrón DTO (Data Transfer Object). La idea es que la API no devuelva directamente las entidades de la base de datos, sino objetos diseñados solo para transportar datos.

1️⃣ El problema de exponer entidades JPA
Si un controller devuelve directamente una entidad:
@GetMapping("/{id}")
public Usuario obtener(@PathVariable Long id) {
    return usuarioService.obtenerPorId(id);
}
pueden ocurrir varios problemas:
se exponen datos sensibles
se muestran relaciones innecesarias
la API queda acoplada a la estructura de la base de datos
